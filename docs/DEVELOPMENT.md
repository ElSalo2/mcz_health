# Разработка и деплой

## Обязательный порядок изменений

Любые изменения вносятся **строго в три этапа**, без пропусков и без изменения порядка:

```
1. Локальная версия  →  2. Git (GitHub)  →  3. Сервер
```

| Этап | Где | Что делается |
|------|-----|--------------|
| **1. Локально** | Рабочая копия на машине разработчика (`catalog_monitor/`) | Правки кода, тесты, проверка |
| **2. Git** | Репозиторий `ElSalo2/mcz_health` на GitHub | `git commit` + `git push` |
| **3. Сервер** | `root@88.218.122.176`, `/opt/mcz_health` | `git pull` / `git reset --hard origin/main` + перезапуск `mcz_health` |

**Запрещено:**
- пушить в Git или деплоить на сервер, минуя локальную проверку;
- менять код сразу на сервере без отражения в локальной копии и Git;
- считать сервер «источником правды» — источник правды всегда **локальная копия → Git**.

## Запрет деплоя без явной команды владельца

**Никто** (включая автоматизацию и AI-ассистентов) **не выполняет** без явного указания владельца проекта:

- `git commit`
- `git push`
- деплой на сервер (`git pull`, `git reset --hard`, `systemctl restart mcz_health`, `scripts/safe_deploy.sh` и т.п.)

Допустимые формулировки команды владельца: «закоммить», «запушь», «задеплой», «выкати на сервер», «перезапусти на сервере».

Пока такой команды нет — изменения остаются **только локально** (этап 1).

## Этап 1: Локальная разработка

```bash
cd catalog_monitor
python -m pytest          # прогон тестов
python -m app.main          # локальный запуск (при необходимости)
```

Перед началом работы убедитесь, что локальная копия совпадает с Git:

```bash
git fetch origin
git status                  # не должно быть незакоммиченных правок, если не ведётся новая задача
git pull --ff-only origin main
```

Если нужно отменить локальные правки и вернуться к версии из Git:

```bash
git restore .
git clean -fd               # осторожно: удаляет неотслеживаемые файлы
```

Конфигурация: `catalog_monitor/.env` (не коммитится). Пример — `.env.example`.

## Этап 2: Git (только по команде владельца)

```bash
git add <файлы>
git commit -m "описание изменения"
git push origin main
```

Проверка после push:

```bash
git log -1 --oneline
git status                  # Your branch is up to date with 'origin/main'
```

## Этап 3: Сервер (только по команде владельца)

Сервер: `root@88.218.122.176`, каталог `/opt/mcz_health`, сервис `mcz_health`.

### Обновление кода

```bash
ssh root@88.218.122.176
cd /opt/mcz_health
git fetch origin
git reset --hard origin/main   # или git pull --ff-only, если нет расхождений
```

### Переменные окружения

Правки `.env` на сервере — только если они уже согласованы и отражены в `.env.example` (или отдельно описаны владельцем). После изменения `.env` нужен перезапуск сервиса.

### Перезапуск

```bash
systemctl restart mcz_health
systemctl is-active mcz_health
```

Скрипт `scripts/safe_deploy.sh` на сервере: pull + restart, но **не перезапускает** сервис, пока идёт товарная проверка. Использовать тоже **только по команде владельца**.

### Проверка после деплоя

```bash
journalctl -u mcz_health -n 30 --no-pager
```

Убедиться, что коммит на сервере совпадает с локальным:

```bash
git log -1 --oneline    # локально и на сервере — один и тот же hash
```

## Краткая памятка

| Действие | Когда |
|----------|--------|
| Править код | Всегда сначала локально |
| `git commit` / `git push` | Только по команде владельца |
| Деплой / restart на сервере | Только по команде владельца |
| Править код на сервере напрямую | **Запрещено** |
