# Режим непрерывного мониторинга

Система **не позволяет пользователям запускать проверки вручную**. Мониторинг выполняется автоматически в фоне.

## Цикл работы

```
┌─────────────────────────────────────────────────────────────┐
│                    Фоновый цикл (24/7)                      │
├─────────────────────────────────────────────────────────────┤
│  1. Скачать фид магазинов                                   │
│  2. Скачать фид товаров                                    │
│  3. Проверить ВСЕ магазины, затем ВСЕ товары из фидов       │
│     (максимум 12 часов на один цикл)                        │
│  4. При критичных нарушениях → алерт пользователям бота    │
│  5. Сразу скачать новый фид и повторить цикл                │
└─────────────────────────────────────────────────────────────┘
```

## Временные ограничения

| Параметр | Значение по умолчанию | Переменная `.env` |
|----------|----------------------|-------------------|
| Максимальная длительность проверки | 12 часов | `MAX_CHECK_DURATION_SECONDS=43200` |
| Резерв на локальные проверки | 10 мин | `LOCAL_CHECK_RESERVE_SECONDS=600` |
| Пауза между HTTP-проверками URL | 0,25 с | `HTTP_URL_SLOT_SECONDS=0.25` |

После завершения проверки (успешно или по таймауту) система **сразу скачивает новый фид** и начинает следующий цикл без паузы.

## HTTP-троттлинг (защита от перегрузки сайта)

HTTP-проверки URL выполняются **строго последовательно**, без параллельных запросов.

1. После парсинга обоих фидов собираются все URL для HTTP-проверки:
   - страницы товаров;
   - изображения товаров (если `CHECK_PRODUCT_IMAGES=true`);
   - страницы магазинов (`url`, `info-page`);
   - фотографии магазинов (если `CHECK_STORE_IMAGES=true`);
   - соцсети (если `CHECK_SOCIAL_LINKS=true`).
2. Пауза между URL задаётся в `HTTP_URL_SLOT_SECONDS` (по умолчанию 0,25 с).
3. `MAX_CHECK_DURATION_SECONDS` — только потолок цикла, на интервал не влияет.

```
интервал_между_URL = HTTP_URL_SLOT_SECONDS
```

4. Каждый URL открывается по очереди; после запроса система ждёт оставшуюся часть интервала.

**Пример для текущих фидов (~47 000 URL, `HTTP_URL_SLOT_SECONDS=0.25`):**

- Интервал: **0,25 с/URL** (~4 HEAD-запроса/сек)
- HTTP-этап цикла: ~47 000 × 0,25 ≈ **11 750 с (~3,3 ч)**
- Потолок цикла: 12 ч (`MAX_CHECK_DURATION_SECONDS=43200`) — запас на медленные ответы и локальные проверки

`http_total_planned` = страницы + изображения. Если один и тот же URL картинки встречается у нескольких товаров, он проверяется по разу на каждый товар — в отчёте это видно как «+N повторных URL» и процент выше 100%.

**При `HTTP_URL_SLOT_SECONDS=0.4`:** ~5,2 ч на HTTP-этап (~2,5 запроса/сек).

Дубликаты URL (внутри фида и между фидами) учитываются один раз.

## Плановое окончание в отчёте бота

В «Последняя проверка» и «История» для выполняющейся проверки показывается строка **плановое окончание**. Это **оценка**, а не жёсткий лимит `MAX_CHECK_DURATION_SECONDS`.

Расчёт опирается на тот же троттлинг, что и реальные HTTP-запросы:

1. После парсинга обоих фидов считается `cycle_http_url_count` — все URL для HTTP (страницы и изображения товаров, страницы и фото магазинов, соцсети при `CHECK_SOCIAL_LINKS=true`).
2. Сохраняется `http_slot_seconds = HTTP_URL_SLOT_SECONDS`.
3. Для каждого фида в статистику пишется `http_total_planned` — число его URL из того же набора.
4. **До начала HTTP** (прогресс 0):  
   `плановое_окончание = начало_проверки_фида + доля_локальных_проверок + http_total_planned × http_slot_seconds`,  
   где доля локальных проверок = `LOCAL_CHECK_RESERVE × (http_total_planned / cycle_http_url_count)`.
5. **Во время HTTP** (есть `http_total_checked`):  
   `плановое_окончание = сейчас + (http_total_planned − http_total_checked) × http_slot_seconds`.

Пример при `HTTP_URL_SLOT_SECONDS=0.25` (~47 000 URL в цикле, ~10 800 URL у товарного фида):

- весь цикл HTTP: ~47 000 × 0,25 с ≈ **3,3 ч**;
- только товарный фид: ~10 800 × 0,25 с ≈ **45 мин** от старта товарной проверки;
- при 2 800 из 10 800 проверенных: ~8 000 × 0,25 с ≈ **33 мин** от текущего момента.

Если фид не изменился (`skip_http`), HTTP пропускается — плановое окончание не показывается.

Поля в `stats_json` проверки: `http_slot_seconds`, `planned_duration_seconds`, `http_total_planned`, `http_total_checked`, `warnings_by_type`, `critical_by_type`.

В финальном отчёте бота после строки «Результат» выводится расшифровка предупреждений и критических проблем по типам (`message_key`).

## Что проверяется

- **Все товары** из фида товаров (`PRODUCT_FEED_URL`)
- **Все магазины** из фида магазинов (`STORE_FEED_URL`)

Проверка включает три этапа:

1. Доступность XML
2. Структура XML
3. Содержимое (поля, URL, изображения — в зависимости от `CHECK_MODE`)

## Уведомления

Политика задаётся списками `message_key` в `alert_policy.py`. Шаблоны — в `app/locales/messages.py`.

### CRITICAL → Telegram

`XML_UNAVAILABLE`, `XML_ERROR`, `PRODUCT_PAGE_UNAVAILABLE`, `PRODUCT_IMAGE_UNAVAILABLE`, `STORE_IMAGE_UNAVAILABLE`, `PRODUCT_MISSING_PICTURE`, `STORE_MISSING_PHOTO`, `PRODUCT_MISSING_CATEGORY`, `PRODUCT_INVALID_CATEGORY`, `PRODUCT_MISSING_NAME`, `PRODUCT_MISSING_PRICE`, `PRODUCT_INVALID_PRICE`, `CATEGORY_INVALID_PARENT`, `DUPLICATE_IDS`

HTTP-алерты — только при ответе **4xx/5xx**.

### WARNING → Telegram

`COUNT_CHANGE`, `FEED_SIZE_CHANGE` (порог `MAX_FEED_SIZE_CHANGE_PERCENT=15`), `STALE_DATA`, `PRODUCT_LOW_PRICE`, `PRODUCT_INVALID_OLDPRICE`

### Без Telegram

`PRODUCT_PRICE_CHANGE`, `PRODUCT_IMAGE_INVALID_CONTENT_TYPE`, `PRODUCT_IMAGE_EMPTY`, `STORE_IMAGE_INVALID_CONTENT_TYPE`, `PRODUCT_MISSING_URL`, `DUPLICATE_URLS`, `PRODUCT_NEGATIVE_STOCK`, `PRODUCT_AVAILABLE_AT_ZERO_STOCK`, прочие предупреждения и пустые поля.

История цен — таблица `product_price_snapshots`. Уведомления об устранении проблем **отключены**.

### Прочие события

| Событие | Кому отправляется |
|---------|-------------------|
| Запрос на доступ к боту | Администратор |

## Роль Telegram-бота

Бот **не управляет** проверками. Он используется для:

- авторизации пользователей;
- просмотра последней автоматической проверки;
- просмотра истории проверок;
- получения алертов о проблемах;
- администрирования пользователей (только админ).

## Компоненты

| Модуль | Назначение |
|--------|------------|
| `ContinuousMonitoringService` | Бесконечный фоновый цикл |
| `CheckOrchestrator` | 3-этапная проверка каждого фида |
| `ResourceChecker` | Последовательная HTTP-проверка URL с троттлингом |
| `UrlThrottlePlanner` | Расчёт интервала между HTTP-запросами |
| `url_collector` | Сбор URL из обоих фидов |
| `NotificationService` | Алерты пользователям по политике `alert_policy` |
| `alert_policy` | Правила, по которым проблема попадает в Telegram |
| `CheckQueryService` | Просмотр результатов через бот |
| `check_formatter` | Форматирование отчётов и оценки планового окончания |
| `check_stats_builder` | Начальная статистика и `planned_duration_seconds` |
